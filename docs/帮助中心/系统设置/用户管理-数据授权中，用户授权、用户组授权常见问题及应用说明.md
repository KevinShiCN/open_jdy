# 2 常见问题

经常有以下几类问题的咨询：

- **场景1**：数据授权，用户组【甲】里设置A供应商，子账号在用户组【甲】里，且单独对子账号用户授权设置B供应商，子账号可以看到供应商A和B。
- **场景2**：数据授权，用户组【甲】设置仓库A，子账号用户授权没有设置仓库，子账号可以看到仓库A的基础资料及单据
- **场景3**：数据授权，用户组【甲】没有设置授权，子账号用户授权设置仓库A，子账号可以看到仓库A基础资料及单据，以及没有仓库的单据。
- **场景4**：数据授权，用户组【甲】设置仓库A，用户组【乙】没设置任何仓库权限，子账号同时在两个用户组里，子账号可以看到仓库A基础资料及单据，以及没有仓库的单据。

以上现象其实都是正常的，而原因都是因为**对子账号同时设置了用户授权和用户组授权，又或者同时设置了多个用户组的权限。**下面来简单说明一下逻辑。

# 3 数据授权的逻辑

- 当同时设置了**用户组和用户授权**（都有设置具体的值），又或者同时设置了**多个用户组**，子账号的权限范围是2个授权结果的合集（取“或”的结果），也就是权限是叠加的
- 当**用户和用户组任意一边里有权限，另一边没有设任何授权**（也就是没有设置任何的值），则以设置了权限的一方的权限为准，用户只能看设置了值的字段的基础资料、单据，以及该字段为空的单据（为空表示不控制，详见下文第5大点）。
- 当**用户组**与用户组之间，任意一组里有权限，另一组没有设任何授权（也就是没有设置任何的的值），则以设置了权限的一方的权限为准，用户只能看设置了值的字段的基础资料、单据，以及该字段为空的单据（为空表示不控制，详见下文第5大点）。

什么是权限叠加？取“或”的结果是什么意思？下面通过不同场景的权限设置演示数据权限的效果。

# 7 实施及应用建议

## **7.1 权限应用场景说明**

| 类型 | 适用场景 | 应用举例 |
| --- | --- | --- |
| 用户授权 | 适用于每个子账号的权限是不一样的，需要按单个用户做数据隔离的场景，适用于公司人员不多，或者需要严格管控个人数据权限的场景 | 深圳的销售员只能看深圳的客户，广州的销售员只能看广州的客户，那么对深圳和广州的业务员个人分别做数据授权下“客户”的授权即可。 |
| 用户授权组 | 适用于同一个岗位的账号权限是一样的，按职能做功能、数据隔离的场景。只要在同一个组别（部门）里，那么大家的查看单据和报表权限是一样的，开单选客户或者供应商或者部门的权限都是一样。适用于公司员工较多，需要按职能进行管理、职能内权限不做区分，一视同仁。 | 销售部可以看所有销售类的单据、报表，同时可以看所有客户的数据。不管是业务员A还是B，大家操作和查看的权限都是一样的对用户组【销售部】做统一的授权，不用对子账号个人授权。 |
| 用户组授权+用户授权同时使用 | 当系统使用人数较多，同时需要按照岗位做比较精细的授权控制时，两种授权方式也可以结合使用。注意：同时使用的前提是，用户个人的权限是包含了用户组本身已有的权限的，也就是用户权限是大于用户组权限的。 | 某个公司有多个分公司，分公司有经理和销售员，则可以先按照分公司进行分组，做用户组授权，为每个分公司设置单独的仓库、客户等数据授权。在用户组授权完成之后，再选择每个用户组的经理，进行用户授权，赋予经理更多的功能权限。 |

>
温馨建议：当子账号个人权限是**小于**用户组的权限时（也就是子账号只拥有部分数据权限），不要同时使用用户组授权+用户授权。

## **7.2 数据为空的影响**

使用数据授权时，建议所设置的字段都维护了对应的值，尽量不要出现为空的值，否则授权不做控制
